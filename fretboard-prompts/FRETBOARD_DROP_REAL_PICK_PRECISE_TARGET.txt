## Root problem

The current moving target looks like a generic blue mechanical gem rather than a guitar pick.

More importantly, the renderer currently asks one oversized visual object to perform two separate jobs:

1. provide a visible, attractive moving target
2. identify an exact string and fret position

That breaks down at higher fret numbers because fret cells become narrower. The large target overlaps adjacent cells and makes the intended fret ambiguous.

## Are we solving the root problem, or just patching the current architecture?

We are solving the root problem.

Do not merely swap the current graphic or reduce its size globally. Separate the decorative moving pick from the precise target/contact coordinate.

The exact string-and-fret coordinate must remain the source of truth. The pick artwork should be visually offset so its pointed tip lands on that coordinate, while a compact marker shows the exact answer location.

## Engineering review

### Option 1: Small patch

Replace the existing gem with the new pick image and make it smaller.

- Complexity: low
- Maintenance: low
- Risk: high-fret ambiguity remains
- Verdict: insufficient

### Option 2: Focused renderer refactor

Keep the current timing, movement, reducer, target generation, and scoring. Introduce a small presentation layer that renders:

- one exact contact point derived from the existing target position
- one decorative guitar-pick image offset from that point
- one compact precision marker centered on that point

- Complexity: low to moderate
- Maintenance: low
- Security/cost: no impact
- Risk: limited to renderer geometry and CSS
- Verdict: recommended

### Option 3: Fully adaptive target-layout system

Dynamically scale, rotate, and reposition the artwork based on viewport, fret width, string spacing, and edge proximity.

- Complexity: high
- Maintenance: higher
- Risk: introduces unnecessary geometry branches
- Verdict: overkill for the current stage

Implement Option 2.

Use Rob Pike's principle: simple data flow, one source of truth, no clever duplicate positioning system.

Use Steve Jobs' principle: the target should feel unmistakably guitar-specific, polished, and obvious without visual clutter.

## Required implementation

### 1. Preserve one timing and movement source of truth

Do not change:

- reducer timing
- `performance.now()` timing domain
- `target.startedAt`
- `target.durationMs`
- `getTargetProgress`
- hit/miss timing
- target spawning
- scoring
- lives
- speed modes
- wrong-answer behavior
- game-over behavior

The current exact target coordinate must remain derived from the existing gameplay state.

Do not introduce:

- a second `requestAnimationFrame` clock
- a CSS duration-based travel animation
- duplicate progress state
- independent visual timing
- delayed visual interpolation

The pick artwork and precision marker must be rendered from the same current target position.

### 2. Introduce a small presentation component

Prefer a compact component such as:

`PickTargetIndicator`

or an equally clear local name.

Its conceptual inputs should be minimal, for example:

- exact target x-coordinate or progress-derived position
- exact target y-coordinate/string position
- active/final-second state if already available
- reduced-motion state if already available

Do not pass the entire game state if a few primitive values are sufficient.

The component should render two layers:

#### Precision layer

A compact marker centered exactly on the intended string/fret target coordinate.

Recommended appearance:

- 5 to 7 CSS pixels at normal desktop scale
- a small dot, short vertical tick, or tiny crosshair-like contact cue
- high contrast against the fretboard
- restrained glow only
- no label
- no large ring
- pointer-events disabled
- does not alter layout

This marker is the authoritative visual answer location.

#### Pick artwork layer

Render the real guitar-pick image as a decorative layer positioned relative to the same contact coordinate.

The pointed tip of the pick must visually land on or immediately adjacent to the precision marker.

Do not center the image on the coordinate.

Use a clear CSS transform or offset so the pick's tip is the anchor. Keep that offset in one place rather than scattering magic numbers across JSX and CSS.

For example, a wrapper may be positioned at the exact coordinate, then the image translated so its pointed tip aligns with the wrapper origin.

The exact transform should be tuned to the actual PNG proportions.

### 3. Size behavior

The current target is too large for narrow high-number fret cells.

Use a modest base size, approximately 20 to 30 percent smaller than the current rendered target.

Do not shrink the pick until it becomes unrecognizable.

Recommended approach:

- use one restrained responsive size with `clamp(...)`
- keep the precision marker constant or nearly constant
- allow the decorative body of the pick to overlap adjacent cells if necessary
- rely on the pointed tip and marker, not the body width, for exact fret identification

Avoid complicated per-fret scaling unless the current renderer exposes fret-cell width so cleanly that a single simple clamp can use it without new state.

Do not introduce a table of special sizes for fret 1, fret 2, fret 10, and so on.

### 4. High-fret clarity

At frets 9 through the highest supported fret:

- the marker must remain centered in the correct fret cell
- the marker must remain centered on the correct string
- the pick tip must point to that marker
- adjacent fret cells must remain visually distinguishable
- the pick body must not be used as the fret boundary indicator

Check open-string placement too. Do not accidentally position the tip outside the visible board.

If edge clipping occurs near the right side, prefer allowing the pick body to extend inward while keeping the marker exact. Do not move the marker away from the true answer coordinate merely to fit the artwork.

### 5. Remove obsolete gem styling

Remove or stop using styling tied only to the old mechanical gem, including obsolete:

- center-hole elements
- vertical stripe elements
- circular shell/housing elements
- oversized target rings
- gem facets that are no longer rendered
- unused keyframes related to the old appearance

Do not perform a broad CSS cleanup. Remove only code made obsolete by this change.

### 6. Interaction and accessibility

The pick and marker must:

- use `pointer-events: none`
- not intercept fretboard clicks
- not change hit testing
- not create a new tab stop
- preserve existing accessible labels and game instructions
- preserve reduced-motion behavior
- include appropriate empty alt text if the image is purely decorative

The precision marker is visual only and should not create redundant screen-reader output.
