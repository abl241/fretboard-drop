PRIMARY GOAL

Fix the original Fretboard Drop horizontal-deadline game so the note pick visibly and smoothly travels from the left side of the stage to the deadline.

The previous single-clock repair did not fix the real browser behavior. The pick still stays near the start position and then pops.

Do not add diagnostics, temporary overlays, instrumentation, console logging, or browser probes. Do not waste time measuring the existing broken visual path.

Instead, replace the fragile presentation mechanism with a simpler platform-native one.

ARE WE SOLVING THE ROOT PROBLEM, OR JUST PATCHING THE CURRENT ARCHITECTURE?

Solve the root problem.

The existing visible travel depends on React repeatedly rerendering small `left` changes from target progress.

That has proven unreliable in the real browser even though unit tests show progress values changing.

The repair should separate responsibilities cleanly:

- React/reducer owns target identity, start time, duration, hit, miss, score, lives, and game state
- CSS owns smooth visual travel from the start position to the deadline over the target's existing duration

Do not reintroduce a second gameplay clock.

ENGINEERING OPTIONS

OPTION 1: KEEP PER-FRAME REACT POSITION UPDATES

Reject.

It has already failed in the real browser.

OPTION 2: NATIVE CSS TRAVEL ANIMATION

Use this.

- one mounted target
- one CSS animation
- duration comes from `target.durationMs`
- browser performs continuous movement
- reducer still decides hits and misses

OPTION 3: IMPERATIVE DOM REQUESTANIMATIONFRAME LOOP

Reject unless CSS animation is genuinely impossible.

It adds more code and another manual animation path.

REQUIRED IMPLEMENTATION

1. OUTER TRAVEL WRAPPER

Refactor the active horizontal pick so an outer wrapper controls horizontal travel.

The outer wrapper must:

- mount once per target
- use `key={target.id}` or equivalent stable target identity
- begin at the configured pick start percentage
- end at the configured deadline percentage
- animate linearly
- use the target's existing `durationMs`
- use CSS animation fill mode so it starts and ends predictably

Do not update the wrapper's `left` or transform every frame from React.

2. INNER VISUAL PICK

Keep the existing pick shape, note text, glow, and final-second styling on an inner element.

Travel animation and decorative animation must be on different DOM layers.

Required separation:

- outer element: horizontal travel only
- inner element: pick body, glow, final-second pulse, correct-resolution appearance

This prevents one CSS `animation` declaration from replacing another.

3. CSS CUSTOM PROPERTIES

Pass only stable per-target values through CSS custom properties, such as:

- `--pick-start-percent`
- `--pick-end-percent`
- `--pick-duration-ms`

Example direction:

```tsx
style={{
  "--pick-start-percent": `${PICK_START_RIGHT_PERCENT}%`,
  "--pick-end-percent": `${DEADLINE_RIGHT_PERCENT}%`,
  "--pick-duration-ms": `${target.durationMs}ms`,
  width: sizePx,
  height: sizePx,
  color: accent.color,
} as React.CSSProperties}
```

Use the repository's TypeScript style conventions.

4. NATIVE TRAVEL ANIMATION

Add a dedicated keyframe:

```css
@keyframes horizontal-deadline-travel {
  from {
    left: var(--pick-start-percent);
  }

  to {
    left: var(--pick-end-percent);
  }
}
```

Apply it only to the active travel wrapper:

```css
.horizontal-deadline-pick-travel--active {
  animation: horizontal-deadline-travel var(--pick-duration-ms) linear both;
}
```

Alternative use of `transform: translateX(...)` is acceptable if it is simpler and preserves exact start and deadline geometry.

Do not use a transition.

Do not feed a new target position every frame.

5. RESOLVED AND MISS STATES

Do not apply active travel animation to:

- resolved-correct snapshot
- miss impact fragments
- timeout or miss reveal elements

Those should remain positioned from their existing resolved progress snapshot or deadline location.

6. GAMEPLAY REMAINS REDUCER-DRIVEN
