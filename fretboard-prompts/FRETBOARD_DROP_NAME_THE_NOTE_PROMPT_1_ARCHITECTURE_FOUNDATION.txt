Required architecture

1. Canonical fretboard target

Create or confirm one canonical target model suitable for both:

- current Fretboard Drop
- future Name the Note

It must provide:

- stable target key
- stable string identity
- string index when useful for rendering
- fret number
- canonical note or pitch class

Conceptually:

```ts
type FretboardTarget = {
  targetKey: string;
  stringId: string;
  stringIndex: number;
  fret: number;
  note: NoteName;
};
```

Adapt this shape to the project’s existing type conventions.

Rules:

- fret 0 represents an open string in the data model
- target identity must not depend on CSS classes, DOM nodes, or pixel coordinates
- target keys must be deterministic
- do not create a second incompatible target model
- preserve current persisted identifiers when practical
- add a compatibility adapter instead of a broad migration when that is simpler and safer

2. Shared eligible-target generation

Create or confirm one shared pure pipeline that can produce valid fretboard targets from settings.

It must support:

- selected strings
- fret minimum
- fret maximum
- selected notes
- inclusion or exclusion of open strings
- current tuning and note naming rules

The expected flow is:

```text
all canonical fretboard positions
→ apply selected-string filter
→ apply fret-range filter
→ apply selected-note filter
→ apply open-string rule
→ return eligible targets
```

Requirements:

- keep the function pure where practical
- preserve deterministic output where the current game depends on it
- keep random target selection separate from pool generation
- do not embed UI or scoring logic
- do not duplicate this filter pipeline for the future mode
- preserve current game behavior exactly

3. Current game compatibility

Update the current Fretboard Drop implementation only as needed to consume the canonical target and shared target pool.

Do not change:

- gameplay direction
- score behavior
- question timing
- session timing
- streak behavior
- target frequency
- weak-area behavior
- results
- Play Again
- filters
- saved settings
- current visuals

If exact behavior cannot be preserved through the proposed extraction, stop and choose a smaller compatibility layer.

4. Open-string semantics

Represent open strings semantically as fret 0.

Do not implement the new separate OPEN visual zone yet.

Confirm that:

- fret 0 can be generated and filtered correctly
- excluding open strings removes fret 0
- including open strings allows valid open-string targets
- no existing behavior is broken

5. Stats and fluency preservation

Do not redesign statistics or fluency in this prompt.

Locate and document:

- current attempt record shape
- current fluency formula location
- current weak-target selection location
- persistence keys and versions

Only make changes required to preserve compatibility with the canonical target.

Do not change existing fluency outputs.

6. Renderer boundary

Do not redesign the fretboard renderer.

Only make a minimal boundary improvement if the current renderer directly owns or invents target identity.

The renderer should receive semantic target information rather than calculate the target from display coordinates.

Do not add:

- realistic wood styling
- new target marker
- countdown visual
- open-string visual zone
- correct, incorrect, or timeout animations

Technical-debt guardrails

- Do not clone the current game.
- Do not build a generalized game plugin framework.
- Do not create speculative abstractions for future modes.
- Do not add backend, auth, database, PostHog, feature flags, or session replay.
- Do not alter unrelated GuitarRise modes.
- Do not change the current visual design.
- Do not rename broad groups of files without need.
- Do not introduce provider-specific dependencies.
- Do not scatter note calculation across UI components.
- Do not place filtering logic inside React rendering.
- Do not use target display coordinates as domain identity.
- Prefer a small shared utility and compatibility adapter over a broad refactor.
- Keep types explicit and code easy to trace.
- Remove dead code only when it is directly replaced by this work.
- Do not add comments that merely restate the code.
