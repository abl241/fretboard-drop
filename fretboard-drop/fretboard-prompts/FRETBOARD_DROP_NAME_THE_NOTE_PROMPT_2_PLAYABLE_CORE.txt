Objective
Build the first playable Name the Note mode on top of the shared target foundation created in Prompt 1.

The mode should:

- highlight one exact semantic fretboard position
- ask the player to identify the note
- let the player answer with note-name buttons
- run for a fixed session duration
- award score and maintain streak or combo
- handle correct, incorrect, and timeout outcomes
- show a basic results screen
- preserve the current Fretboard Drop game unchanged

This prompt should produce a complete playable vertical slice, not final visual polish.

Implement:

4. note-answer controls

6. per-question timeout

8. basic feedback
9. basic results
10. Play Again

- 0–1000 Name the Note fluency
- weak-area practice
- PostHog

- mastery heat map

- backend, auth, database, or multiplayer

Architecture requirements

1. New mode identity

Add a clear mode identity for Name the Note.

Use the smallest existing routing or mode-entry pattern.

Do not create a separate app.

Do not build a generalized mode plugin system.

The user should be able to reach:

- existing Fretboard Drop
- new Name the Note

without disrupting the current entry flow.

2. Shared settings

Reuse the current practice settings where practical:

- selected strings
- selected notes
- fret range
- include open strings
- session duration if already configurable

Do not duplicate settings state if a shared settings model already exists.

If the existing settings UI is tightly coupled to Fretboard Drop, create the smallest shared adapter or wrapper needed.

Do not redesign the settings screen in this prompt.

3. Shared target pool

Use:

src/lib/fretboardTargets.ts

for all Name the Note eligible-target generation.

Do not recalculate notes in the component.

Do not introduce a second target type.

The Name the Note mode must consume canonical FretboardTarget objects.

4. Exact-position prompt

Display one exact string-and-fret position.

Requirements:

- one specific string row
- one specific fret cell
- target remains unambiguous
- marker fits within the exact cell
- marker does not span multiple frets
- marker does not highlight the whole string
- marker does not use the whole fret column as the only cue

For now, use a simple precise marker.

Do not implement the polished wireframe fretboard yet.

5. Open-string handling

Open strings are fret 0 in the data model.

For this prompt, implement the minimum clear visual treatment:

- open target appears outside the numbered fretted area
- it is aligned with the correct string
- it is visually distinct from fret 1
- it is clearly labeled OPEN

Do not build the final polished OPEN zone yet.

6. Session structure

Use a fixed-duration score run.

Default to 60 seconds unless the current app already has a shared compatible setting.

Requirements:

- no lives
- session timer counts down
- player answers as many questions as possible
- run ends when session time reaches zero
- results screen appears
- Play Again restarts with the same settings

7. Per-question timing

Each target has a limited answer window.

Use a simple initial timing rule compatible with the roadmap, approximately 2 to 6 seconds.

Do not finalize adaptive timing in this prompt.

Represent timing explicitly:

- totalQuestionMs
- remainingQuestionMs
- countdownFraction

Use:

```ts
countdownFraction = remainingQuestionMs / totalQuestionMs;
```

Even if the full visual countdown is deferred, the state must be available for Prompt 5.

Use normalized percentage, not fixed-second visual bands.

8. Note-answer controls

Show large note buttons.

Minimum:

- A
- B
- C
- D
- E
- F
- G

Requirements:

- clicking a button answers immediately
- add keyboard shortcuts when practical
- buttons remain easy to scan
- selected-note configuration must still control target generation
- follow the project’s current note naming convention
- do not introduce a complex enharmonic-preference system

If the selected target is accidental under the current note model, use the simplest existing supported answer naming behavior. Do not invent a second note system.

If the current project cannot support accidentals cleanly with A–G buttons, constrain the playable first version to natural notes only and document this explicitly, but preserve the underlying selected-note settings for later work.

Prefer supporting the current note model if it is already straightforward.

9. Answer evaluation

Use canonical target.note.

Use explicit outcomes:

```ts
type QuestionOutcome =
  | "idle"
  | "correct"
  | "incorrect"
  | "timeout";
```

Correct:

- award score
- continue or increase streak
- show brief correct feedback
- advance quickly

Incorrect:

- record selected wrong answer in transient state
- reveal correct note
- reset or update streak consistently
- advance after a short pause

Timeout:

- do not treat any answer button as selected
- reveal correct note
- reset or update streak consistently
- advance after a short pause

Avoid long pauses.

10. Score and streak

Reuse the existing score philosophy where practical.

Do not copy the current lives-based gameplay model.

For this first version:

- correct answers earn points
- faster correct answers may earn more
- streak or combo may multiply or supplement score
- wrong answers and timeouts do not award points
- scoring must be easy to understand
- keep the implementation isolated so Prompt 3 can align it with fluency

Do not change current Fretboard Drop scoring.

11. Results

Add a basic Name the Note results screen.

Show at minimum:

- final score
- correct answers
- incorrect answers
- timeouts
- accuracy
- average correct response time
- best streak
- Play Again
- Back or Exit

Do not add advanced fluency, weak-area, or adaptive next-action logic yet.

12. Persistence

Persist only what is necessary for the playable core:

- best score for Name the Note
- last-used settings if the shared settings system already does this

Use mode-specific keys.

Do not write Name the Note results into current Fretboard Drop keys.

Do not add full attempt history or fluency persistence yet unless the current architecture requires a minimal attempt record.

13. Renderer and state boundaries

Keep game state separate from rendering.

The renderer should receive semantic state such as:

- target
- countdownFraction
- outcome
- revealedCorrectNote
- interactionEnabled

Do not calculate note identity or score inside the renderer.

Do not make gameplay logic depend on CSS classes or DOM measurements.

14. Current game protection

Do not change:

- existing Fretboard Drop behavior
- lives
- current target distribution
- current score
- current combo
- current timing
- current results
- current analytics
- current weak-area mode
- current persistence
- current visuals

If shared extraction is needed, preserve behavior through tests.
