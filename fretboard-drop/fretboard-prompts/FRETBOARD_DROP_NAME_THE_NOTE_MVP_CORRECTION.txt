RANDOMIZED TARGETS, LARGER FRETBOARD, ANSWER-BUTTON COUNTDOWN, AND RETRY AFTER WRONG ANSWERS

Context

The playable Name the Note mode works, but manual testing exposed four MVP problems:

1. Target selection is deterministic, so runs repeat the same sequence and appear to move systematically through strings/frets.
2. The fretboard is too small and does not match the approved wireframe proportions.
3. The most recent correction made the fretboard itself change colors as the per-question countdown. That is no longer wanted.
4. A wrong answer currently reveals the correct answer and ends the question. The desired behavior is to let the player try again while time continues.

Keep this as a focused MVP correction. Do not add elaborate effects, new systems, or the later realistic fretboard renderer.

ROOT CAUSE ALREADY IDENTIFIED

The current selector is deterministic:

```ts
function selectNameTheNoteTarget(
  pool: readonly FretboardTarget[],
  seed: number,
  previousTargetKey?: string,
): FretboardTarget {
  const index = Math.abs(seed * 7 + 3) % pool.length;
  ...
}
```

The seed starts at 0 every run and increases predictably, so every run follows the same mathematical sequence over a canonically ordered pool.

Replace this approach. Do not merely change the multiplier or offset.

OBJECTIVE

Deliver a corrected Name the Note MVP with:

- genuinely randomized but well-distributed targets
- a larger, wireframe-aligned fretboard
- no fretboard color-changing countdown
- draining color inside the A–G answer buttons
- a small numeric per-question countdown
- retry after wrong answers without revealing the correct answer
- existing run duration, score framework, settings, results, persistence, and original Fretboard Drop behavior preserved unless explicitly changed below

1. REPLACE DETERMINISTIC TARGET SELECTION WITH A SHUFFLED DECK

Use a simple shuffled-deck model.

Required behavior:

- Build the eligible target pool from the current settings.
- Shuffle the pool with Fisher-Yates at the beginning of each run.
- Draw targets one by one from the shuffled deck.
- When the deck is exhausted, reshuffle the same eligible pool.
- Prevent the first target of a newly shuffled deck from matching the immediately previous target when more than one eligible target exists.
- Every new run should use a newly shuffled order.
- Do not allow a deterministic fixed sequence that restarts identically each run.
- Preserve exact target identity through `targetKey`.
- Preserve OPEN targets as fret 0 in the separate OPEN zone.

Implementation guidance:

- Remove `targetSeed` if it is no longer needed.
- Replace `selectNameTheNoteTarget(pool, seed, ...)` with a small pure shuffle/deck helper.
- Keep the solution simple and local to Name the Note unless an existing generic shuffle helper is already appropriate.
- Accept an optional RNG function in the pure shuffle helper for deterministic tests, for example:
  `shuffleTargets(targets, rng = Math.random)`.
- Do not add a library for shuffling.
- Do not use crypto APIs.
- Do not introduce a broad game-engine abstraction.

Tests must prove:

- a run draws from a shuffled deck rather than the old arithmetic sequence
- every eligible target appears once before the deck repeats
- immediate duplicates are prevented across a deck reshuffle when possible
- a deterministic injected RNG makes tests stable
- starting a new production run does not intentionally reset to the same fixed order

Preserve `countdownFraction` as semantic game state because the answer-button fill will use it.

3. SHOW QUESTION TIME BY DRAINING COLOR INSIDE THE ANSWER BUTTONS

Use the seven A–G answer buttons as the primary visual countdown.

Required behavior:

- Each answer button has a stable letter that remains fully readable throughout.
- A colored fill sits behind the letter.
- The fill begins full at the start of each question.
- The fill drains vertically and continuously as `countdownFraction` decreases.
- The same countdown fraction should be shown consistently across all currently available answer buttons.
- The letter must stay above the fill with strong contrast.
- Button size, letter position, and button outline remain stable.
- Keep a small numeric per-question countdown as a secondary cue.
- The numeric countdown should represent the current question, not only the total 60-second run.
- Retain the existing total-run timer in the top bar if already present, but clearly distinguish it from the small question countdown.

Preferred simple implementation:

- one absolutely positioned fill layer inside each answer button
- fill height or scale derived from `countdownFraction`
- `transform-origin: bottom` or a clear top-to-bottom drain
- letter rendered in a higher z-index
- no canvas, SVG animation framework, or particle system

The drain must remain visible even when a wrong answer has been attempted.

4. WRONG ANSWER MUST ALLOW ANOTHER TRY

Change wrong-answer behavior.

Current unwanted behavior:

- wrong answer sets terminal `outcome: "incorrect"`
- correct note is revealed
- question advances after a delay

Required behavior:

- A wrong answer does not end the question.
- Do not reveal the correct answer after a wrong guess.
- Keep the same target active.
- Keep the question countdown running without resetting.
- Allow the player to choose another answer immediately.
- Mark the attempted wrong letter red or otherwise clearly incorrect.
- Disable that wrong letter for the remainder of the current question so the player does not repeat it accidentally.
- Other answer buttons remain enabled.
- Support more than one wrong attempt on the same question.
- Clear the attempted-wrong-answer set when the next target begins.

Recommended data model:

- replace the single terminal wrong-answer state with something like:
  `wrongAnswers: readonly Note[]`
- keep terminal outcomes only for:
  - idle/in-progress
  - correct
  - timeout
- If retaining `incorrect` as an outcome for compatibility, it must not be terminal and must not trigger answer reveal or question advance. Prefer the simpler clean model.

Scoring and streak:

- Incorrect attempt count should increment for each wrong button press.
- Do not award points for wrong attempts.
- Do not reveal the correct answer.
- Keep the current streak behavior as simple as possible:
  - a wrong attempt resets the current streak
  - a later correct answer on the same target may still earn base/speed points
  - after one or more wrong attempts, do not increment the streak for that target
- Avoid inventing a complicated penalty system.
- Correct on first attempt continues/increments streak as before.
- Timeout resets streak.

Correct answer:

- marks the selected correct button green
- awards score
- advances after the existing brief correct delay

Timeout:

- ends the question
- reveals the correct letter only after time reaches zero
- advances after the existing timeout/reveal delay
- no wrong answer should be falsely shown as selected
