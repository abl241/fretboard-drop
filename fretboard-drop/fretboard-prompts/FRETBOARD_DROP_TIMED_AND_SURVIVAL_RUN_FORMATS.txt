## Product goal

Offer both current run-ending styles:

1. **Timed Trial**
   - Fixed 60-second run
   - Misses count against accuracy and reset the combo
   - Misses do not end the run
   - This is the default for first-time users

2. **Survival**
   - Three missed targets end the run
   - No 60-second countdown or time-based ending
   - This is the higher-pressure arcade option

The user should choose the run format from the existing Fretboard Drop start screen. The most recently selected format should persist locally.

## Root-problem review

Explicitly answer in your final report:

**Are we solving the root problem, or just patching the current architecture?**

The root problem is that practice-oriented users and arcade-oriented users need different run-ending rules. The current implementation silently combines both rules, because every run ends at 60 seconds or after three misses. The solution is to make the ending contract explicit and use one shared game engine with a format-specific completion condition.

## Architecture review

Before implementation, compare these options briefly in the final report:

### Option 1: Small patch to current setup
Add conditional checks directly around `timeLeftMs` and `lives`.

- Lowest code change
- High risk of scattered format checks
- Easy to accidentally show the wrong HUD metric or mix score histories

### Option 2: Simple shared architecture, recommended
Add one explicit run-format type and store it in game state. Keep one reducer and one rendering flow, with small helper functions for format-specific behavior.

- Low complexity
- Clear maintenance
- No duplicated game engines or pages
- Easy to test
- No backend, database, or new deployment

### Option 3: Strategy-object or separate-engine architecture
Create separate timed and survival engines or generalized rule objects.

- More scalable in theory
- More abstractions and moving parts
- Overkill for two simple formats at the current product stage

Implement Option 2 unless repository evidence reveals a strong reason not to.

## Required implementation

### 1. Add explicit run-format state

Add `DropRunFormat` to the appropriate types.

The selected format must be present in the active `DropGameState` and completed `DropGameResult`, so the reducer, HUD, results, replay, analytics, and persistence all use the format that actually started the run.

The run format must be captured at start. Changing setup state later must not alter an active or completed run.

### 2. Add local preference persistence

Add a small localStorage preference helper in the existing utility style.

Suggested key:

```ts
fretboard-drop:run-format:v1
```

Requirements:

- New users default to `"timed"`.
- Returning users without this key also default to `"timed"`.
- Invalid stored values fall back safely to `"timed"`.
- Selecting a format persists immediately.
- Failure to access localStorage must not break the game.

Do not add a new persistence library.

### 3. Add a compact start-screen selector

Place a clear **Run Type** selector near the existing Pick Speed selector. Use the existing visual language and component patterns rather than inventing a new settings screen.

Create a small reusable selector component, similar in spirit to `SpeedModeSelector`.

Each option must show:

- clear label
- one-line explanation
- selected state using `aria-pressed`

Timed Trial must appear first and be the default.

Update the start metadata pills:

- Timed Trial: show `60 seconds`
- Survival: show `3 misses`
- Keep `frets 0-11`
- Keep the context-specific best display, but it must reflect the selected run format

Do not show both `3 lives` and `60 seconds` as if both end conditions apply.

Use Krug's rule: the user should understand the difference without opening help.

Use Tufte's rule: do not add decorative text or duplicate explanations.

Use Jobs' rule: make the choice feel intentional and polished, not like an engineering toggle.

### 4. Make reducer completion rules format-specific

#### Timed Trial

- Compute and display the existing 60-second countdown.
- Complete when `timeLeftMs <= 0`.
- On a missed target:
  - increment `misses`
  - reset combo
  - show the existing miss reveal/cue
  - remove the missed target
  - do not decrement lives as a gameplay-ending resource
  - after the reveal, spawn the next target and continue if time remains
- The run must not end because misses reached three.

#### Survival

- Do not complete because 60 seconds elapsed.
- On a missed target:
  - decrement remaining lives
  - increment `misses`
  - reset combo
  - use the existing miss reveal/cue
  - the third miss completes the run after the existing reveal
- Wrong fret taps must continue not to remove lives.
- The run can continue beyond 60 seconds.

Keep `performance.now()` as the single gameplay clock. Do not introduce another timer or interval.

Avoid duplicating the reducer. Prefer small explicit helpers such as:

- whether the format uses a timer
- whether a miss removes a life
- whether the current state should complete

Do not build a generic rules framework.

### 5. Make the HUD format-specific

Accessibility:

- Prefer read fallback/migration behavior consistent with the current speed-mode legacy handling.

Do not mix Timed Trial and Survival values in `Best`, `raw best`, trend strips, or results motivation.

Review all helper signatures that currently accept string selection, practice context, and speed mode. Thread run format through only where required for correct scoping.

## Design and engineering principles

### Steve Krug

- The difference between the two choices must be self-evident.
- Use familiar labels and visible selected states.
- Keep the primary Start Run action obvious.
- Do not make users infer whether misses or time ends the run.

### Edward Tufte

- Show only the metric that governs the current run.
- Avoid redundant timer-plus-lives display.
- Preserve information density without visual noise.
- Keep result comparisons honest by separating incompatible score contexts.

### Steve Jobs

- The feature should feel like two purposeful experiences, not a debug setting.
- Preserve the polished first-minute experience.
- The start screen should still have one obvious path into play.
- Do not clutter the app with advanced configuration.

### Rob Pike and Ken Thompson

- Keep one game engine.
- Make state and end conditions explicit.
- Prefer small pure helpers and straightforward branching.
- Avoid duplicated reducers, generalized rule engines, unnecessary hooks, or new abstractions.
- Do not overcomplicate.
